Compile time : It is when the compiler translates the source code into machine code and generates an executable file.
It finds syntax errors, type errors, and it also knows at this time how much memory to allocate for a variable.

Runtime : It is when the executable file is executed. It finds errors that the compiler could not find by just looking
at the source code. e.g. Dividing by 0, segmentation fault, etc.

Compiler : It read the whole source code first and builds a machine code and then executes it
Interpreter : It read and executes code line by line

Preprocessor : It is a tool that runs immediately before the compiler runs. It scans the source code for preprocessor
directive (every line starting with a '#'). These preprocessor directives gives commands to the preprocessor
e.g. #include <iostream>, tells the preprocessor to include the iostream library. So, the preprocessor copies and pastes
the code from the library with our code and sends the extended file(translational unit) to the compiler.
The compiler never knows that the code ever had #include <iostream> in it.

Namespace : It is the way to group the code together and avoid name collisions.
e.g. the 'std' namespace has all the code for us to use things like cout, cin, vector etc. We use the namespace by using
the prefix :: which is called the scope resolution operator. Writing std::cout, std::cin everytime is a tedious task
so we use 'using namespace std' which tells the compiler that if it sees something like cout and does not know what it
is it needs to check the std family

We can make our own namespaces
namespace GamePhysics{

}

namespace User{

}

we have same variable names but different meaning of usage.

cout << User::gravity << endl;

How are data types stored in memory
e.g. int -> it is of 4 bytes -> 32 bits
our RAM has group of 8 switches (8 Bits)
so, int allocates 4 bytes of space in the memory
our machine can't understand the integer value so it converts it to binary code, that the computer can understand.
e.g. 5 is converted to 101
as the compiler can't shrink the memory allocated, it puts the binary values towards the end of the 32 bits.
and the leftover bits are filled with zeroes.

Negative numbers are stored in memory by using 2's complement
it takes the negative number, converts it to binary by ignoring the sign
then we take the 1's complement of the binary code (flip the bits)
and then we add 1 that is the 2's complement.
e.g. -5 -> 5 -> 101
00000000 00000000 00000000 00000101
11111111 11111111 11111111 11111010
11111111 11111111 11111111 11111011 -> this is how -5 is stored in memory
the most significant bit tells if the number is positive or negative
1 -> negative
0 -> positive

How are these negative numbers printed if they are stored as '11111111 11111111 11111111 11111011' in the memory?
it checks the msb, and it knows that it is a negative number
after that it does the reverse 2's complement
11111111 11111111 11111111 11111011
11111111 11111111 11111111 11111010 subtract -1
00000000 00000000 00000000 00000101 flip the bits
converts it into base 10 -> 5 and adds the minus sign and returns -5
